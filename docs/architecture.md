## 2. 数据流向图

```
┌─────────┐     ┌─────────────┐     ┌─────────────────┐
│ 用户提问 │────▶│  "查询最近   │────▶│  Embedding 编码  │
│         │     │  一周的订单" │     │                 │
└─────────┘     └─────────────┘     └────────┬────────┘
                                             │
                                             ▼
┌──────────────────────────────────────────────────────────┐
│                    向量检索 (Top-K=3)                      │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐  │
│  │ orders 表   │  │ users 表    │  │ products 表     │  │
│  │ 相似度: 0.92 │  │ 相似度: 0.75 │  │ 相似度: 0.45    │  │
│  │ 描述: 订单表 │  │ 描述: 用户表 │  │ 描述: 商品表    │  │
│  └──────┬──────┘  └──────┬──────┘  └─────────────────┘  │
└─────────┼────────────────┼──────────────────────────────┘
          │                │
          ▼                ▼
┌──────────────────────────────────────────────────────────┐
│              获取完整表信息 (文本存储)                      │
│                                                          │
│  orders 表完整信息:                                        │
│  ├── 字段: order_id, user_id, amount, status, created_at │
│  └── 说明: 订单金额单位为分，status: 0-待支付 1-已支付      │
│                                                          │
│  users 表完整信息:                                         │
│  ├── 字段: user_id, username, email, phone, created_at   │
│  └── 说明: 用户唯一标识为 user_id，与 orders 一对多关系     │
└────────────────────────┬─────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────┐
│              Agent 1: 字段提取专家                        │
│                                                          │

                         │
```

## 4. 关键设计决策说明

### 4.1 为什么表描述向量化，字段信息不向量化？

```
用户问题: "查询最近一周的订单"

方案 A: 全部向量化
┌────────────────────────────────────────┐
│ 向量检索结果:                           │
│ • orders.created_at (相似度 0.95)      │
│ • orders.order_id (相似度 0.30)        │
│ • users.user_id (相似度 0.25)          │
│ • users.created_at (相似度 0.88) ◄── 噪音！│
└────────────────────────────────────────┘
问题: 丢失表级别上下文，容易混入无关字段

方案 B: 表描述向量化 + Agent 提取
┌────────────────────────────────────────┐
│ 向量检索结果:                           │
│ • orders 表 (相似度 0.92)              │
│ • users 表 (相似度 0.75)               │
│                                        │
│ Agent 1 分析:                          │
│ • "最近一周" → 需要 created_at         │
│ • "订单" → 需要 orders 表              │
│ • 关联用户 → 需要 users 表             │
│ • 输出完整字段列表，不遗漏              │
└────────────────────────────────────────┘
优势: 保留完整上下文，提取更精准
```

### 4.2 为什么使用两个 Agent？

```
单 Agent 方案:
用户问题 ──▶ [检索表信息] ──▶ [大模型生成SQL]
                              输入: 用户问题 + 多表完整信息
                              问题: 上下文过长，容易遗漏关键字段

双 Agent 方案:
用户问题 ──▶ [检索表信息] ──▶ [Agent 1 提取字段] ──▶ [Agent 2 生成SQL]
                              输入: 完整信息           输入: 精简信息
                              输出: 相关字段           输出: SQL
                              优势: 分阶段处理，每阶段专注一个任务
```
